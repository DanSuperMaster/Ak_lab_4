## Язык программирования

Lisp-подобный язык на S-выражениях. Поддерживает рекурсивные функции, ветвления,
циклы, арифметику, строки и потоковый ввод-вывод.

### BNF

```text
<program> ::= { <top-level-form> }
<top-level-form> ::= <defun> | <expr>

<expr> ::= <integer> | <string> | <identifier> | 'T' | 'NIL'
            | <setq> | <if> | <while> | <progn> | <cond>
            | <arith> | <compare> | <io> | <list> | <call>

<integer> ::= [ '-' ] <digit> { <digit> }
<string> ::= '"' { <любой символ кроме '"'> | '""' } '"'
<identifier> ::= <id-char> { <id-char> }   

<setq> ::= '(' 'SETQ' <identifier> <expr> ')'
<if> ::= '(' 'IF' <expr> <expr> [ <expr> ] ')'
<while> ::= '(' 'WHILE' <expr> <expr> ')'
<progn> ::= '(' 'PROGN' { <expr> } ')'
<cond> ::= '(' 'COND' { '(' <expr> <expr> ')' } ')'

<arith> ::= '(' ( '+' | '-' | '+C' | '-B' | '*' | '/' | '%' ) <expr> <expr> ')'
<compare> ::= '(' ( '=' | '<' | '>' ) <expr> <expr> ')'

<io> ::= '(' 'READ' ')'
            | '(' 'WRITE' <expr> ')'
            | '(' 'PEEK'  <expr> ')'

<list> ::= '(' 'LIST' { <integer> } ')'

<defun> ::= '(' 'DEFUN' <identifier> '(' { <identifier> } ')' <expr> ')'
<call> ::= '(' <identifier> { <expr> } ')'
```

### Семантика ключевых слов

| Форма | Значение |
|-------|----------|
| `(SETQ x e)` | вычислить `e`, сохранить в переменную `x` |
| `(IF c a b)` | если `c` ≠ 0 — `a`, иначе `b` (ветка `b` опциональна, по умолчанию `NIL`) |
| `(WHILE c body)` | пока `c` ≠ 0 выполнять `body` |
| `(PROGN e1 … en)` | вычислить по порядку, значение = последнее |
| `(COND (c1 b1) …)` | первая ветвь с истинным условием |
| `(+ - * / %)`, `(= < >)` | бинарные операции; сравнение даёт `1`/`0` |
| `(+C a b)` / `(-B a b)` | сложение с переносом / вычитание с заёмом: учитывают флаг `C` (`+C` = `a+b+C`, `-B` = `a-b-C`) и снова выставляют его — примитивы многословной арифметики |
| `(READ)` | снять один токен из буфера ввода (пустой буфер → останов модели) |
| `(WRITE e)` | вывести младший байт `e` в буфер вывода |
| `(PEEK a)` | прочитать слово памяти данных по адресу `a` |
| `(LIST n…)` | разместить числа `n… 0` в памяти данных, вернуть адрес начала |
| `(DEFUN f (p…) body)` | определить функцию; `(f a…)` — вызов |

### Семантика

- **Стратегия вычислений** —  аргументы вычисляются слева направо до применения функции.
- **Любое выражение возвращает значение** на вершину стека: арифметика, сравнения,
  `IF`, `COND`, вызов функции.
- **Области видимости.** Переменные — глобальные, параметры внутри функций являются локальными. 
- **Типизация** — бестиповая: всё представлено 32-битным машинным словом.  Строки и массивы представлены **адресом** их
  начала в памяти данных.
- **Литералы**: целые (в т.ч. отрицательные), строки (cstr), `T`, `NIL`, массивы
  (`LIST`). <!-- Печать строк и чисел реализована **на самом языке** (функции
  `PRINT-STR` / `PRINT-NUM` в тестах) поверх примитивов `WRITE`/`PEEK`.-->

## Организация памяти

Архитектура **фон Неймовская** - память команд и память данных общая

```text
        Регистры
+-----------------------------+
| PC   счётчик команд         |
| AR   адресный регистр       |
| IR   = opcode : arg(24 бит) |
| C    флаг переноса/заёма АЛУ|
+-----------------------------+

   Стеки (глубина 1024)
+-----------------------------+
| DS  стек данных (операнды)  |
| RS  стек возвратов (PC,     |
|     сохранение пар/локалов) |
+-----------------------------+

   Единая память (фон Нейман), слово 32 бита, адресация по словам
+------------------------------------------------+
| 0           : код (инструкции)                 |  ◄── PC
|   …                                            |
| data_offset : секция данных                    |  ◄── AR
|   c+0 : переменная 1                           |
|   c+1 : переменная 2 …                         |
|   l+0 : строки (cstr) и массивы (LIST)         |
|   …                                            |
+------------------------------------------------+
| 0x7FFFFE : IN_PORT   (memory-mapped input)     |
| 0x7FFFFF : OUT_PORT  (memory-mapped output)    |
+------------------------------------------------+
```

Т.к. стековая архитектура, то присутствуют 2 аппаратных стека, отдельно от основной памяти, длина стеков - 1024 
элемента, DS используется для хранения переменных, для передачи данных в функции, а так же для косвенной загрузки 
(будет ниже). RS используется для сохранения адреса возврата при вызове функции.

**Адресация**

В памяти данных и памяти команд все слова хранятся и выравниваются до 32-битных. (Т.е. ондин инкремент `PC` равен 4 байтам). Адрес всегда кратен 4 байтам. 
Данные всегда размещаются после кода инструкций, для этого высчитывается offset.
Адресация прямая (`LOAD/STORE arg`), так же присутствует и косвенная адресация по вершине стека (`LOAD_IND`)

#### Строки (C-style)

Строковые литералы укладываются побайтово в секции данных: по слову на символ + терминатор `0`; одинаковые строки разделяют память.
Литерал даёт на стек адрес начала

#### Массивы

Построен по аналогии с C-строкой, элементы побайтово кладутся в секции данных: числа подряд + терминатор `0`; литерал даёт адрес начала.

**Размещение объектов (литералы/строки/переменные/процедуры):**

| Объект | Куда попадает                                                                                                             | На каком этапе                                                       | Адресация |
|--------|---------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------|-----------|
| Целочисленный литерал | непосредственно в инструкцию (`PUSH 24` — 24-битный знаковый аргумент)                                                    | во время трансляции слова                                           | непосредственная |
| Строка (cstr) | секция данных (побайтово + терминатор `0`)                                                                                | при первом появлении строки  | адрес начала кладётся на стек (`PUSH` адрес-литерал), к содержимому — косвенная (`LOAD_IND`/`PEEK`) |
| Массив (`LIST`) | секция данных (элементы подряд + терминатор `0`)                                                                          | во время трансляции литерала                                        | адрес начала кладётся на стек (`PUSH` адрес-литерал), к элементам — косвенная (`LOAD_IND`/`PEEK`) |
| Переменная | секция данных (одно слово, инициализируется нулём)                                                                        | при первом появлении под неё выделяется память                      | прямая (`LOAD`/`STORE addr`) |
| Процедура (`DEFUN`) | код в памяти команд; определение «перепрыгивается» инструкцией `JMP`, вызов — `CALL` (адрес возврата → RS), выход — `RET` | во время трансляции (только верхний уровень)                         | прямая (`CALL addr`) |

## Система команд

### Особенности процессора

- **Стековая машина**: операнды берутся с/кладутся на стек данных `DS`; адреса
  возврата — на стеке возвратов `RS`.
- **Типы данных**: 32-битное знаковое целое.
- **Память**: единая, словная адресация.
- **Поток управления**: `JMP` (безусловный), `JZ` (по нулю — АЛУ честно считает
  `0 − TOS`, заём в флаг `C`, переход по `IF_NC`), `CALL`/`RET` через `RS`.
- **Управление**: микропрограммное.

### Кодирование инструкций

Бинарное, одна инструкция — одно 32-битное слово:

```text
 биты 31..24 | биты 23..0
   опкод     | аргумент (знаковый, 24 бита)
```

Все инструкции построены таким образом (первые 8 бит - опкод, остальные 24 - аргумент)

### Набор инструкций

`такты = 2 (выборка: AR ← PC; IR ← MEM[AR], PC++) + число микрокоманд опкода`.

| Opcode | Mnemonic | Operand | Тактов | Действие | Описание |
|:------:|:---------|:-------:|:------:|:---------|:---------|
| 0x01 | `PUSH`     | `imm`  | **3** | `DS.push(imm)` | Положить 24-битную константу на стек |
| 0x02 | `LOAD`     | `addr` | **4** | `DS.push(MEM[addr])` | Прямое чтение из памяти (ввод: `addr = IN_PORT`) |
| 0x03 | `STORE`    | `addr` | **4** | `MEM[addr] = DS.pop()` | Запись в память (вывод: `addr = OUT_PORT`) |
| 0x04 | `LOAD_IND` | –      | **4** | `a = DS.pop(); DS.push(MEM[a])` | Косвенное чтение по адресу с вершины стека |
| 0x10 | `ADD`      | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push((a + b) mod 2³²)` | Сложение по модулю 2³²; `C` ← перенос (старший бит суммы) |
| 0x11 | `SUB`      | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push((a - b) mod 2³²)` | Вычитание по модулю 2³²; `C` ← заём |
| 0x15 | `ADDC`     | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push((a + b + C) mod 2³²)` | Сложение **с переносом**: к сумме прибавляется `C`; снова `C` ← перенос |
| 0x16 | `SUBB`     | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push((a - b - C) mod 2³²)` | Вычитание **с заёмом** по флагу `C` |
| 0x12 | `MUL`      | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push(a * b)` | Умножение (флаг `C` не трогает) |
| 0x13 | `DIV`      | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push(a / b)` | Целочисленное деление (флаг `C` не трогает) |
| 0x14 | `MOD`      | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push(a % b)` | Остаток от деления (флаг `C` не трогает) |
| 0x20 | `EQ`       | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push(a == b)` | Равно: на стек `1`/`0` |
| 0x21 | `LT`       | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push(a < b)` | Меньше: на стек `1`/`0` |
| 0x22 | `GT`       | –      | **3** | `b = DS.pop(); a = DS.pop(); DS.push(a > b)` | Больше: на стек `1`/`0` |
| 0x30 | `JMP`      | `addr` | **3** | `PC = addr` | Безусловный переход |
| 0x31 | `JZ`       | `addr` | **4** | `if DS.pop() == 0: PC = addr` | Переход, если вершина равна нулю |
| 0x32 | `CALL`     | `addr` | **4** | `RS.push(PC); PC = addr` | Вызов функции |
| 0x33 | `RET`      | –      | **3** | `PC = RS.pop()` | Возврат из функции |
| 0x50 | `DUP`      | –      | **3** | `DS.push(DS.peek())` | Дублировать вершину стека |
| 0x51 | `POP`      | –      | **3** | `DS.pop()` | Удалить вершину стека |
| 0x52 | `SWAP`     | –      | **3** | `TOS, NOS = NOS, TOS` | Обменять два верхних элемента |
| 0xFF | `HALT`     | –      | **3** | останов | Остановить тактовый генератор |

## Транслятор

Консольное приложение `utils/compiler.py`.

#### Интерфейс командной строки:

```text
python -m utils.compiler <source.lisp> <output.bin> [--dump <listing.txt>]
```

* Входные данные:
  * `source.lisp` - имя входного файла с исходным кодом на Lisp-подобном языке
  * `output.bin` - имя выходного файла, в нём будет бинарный машинный код для запуска
  * `listing.txt` - имя файла, куда будет записан дамп компиляции (опционально)
* Выходные данные:
    * `output.bin` - бинарный машинный код для запуска
    * `listing.txt` - файл, куда будет записан дамп компиляции (опционально)
    * в `stdout` будет написан результат работы компилятора (сколько строк скомпилировано, название выходного бинарного файла и дамп)

#### Этапы работы:

```text
исходник ─► Lexer ─► токены ─► Parser ─► AST (списки) ─► Compiler ─► (code, data) ─► bin
```

1. **Лексер** (`utils/lexer.py`): токены — числа (с унарным минусом), строки
   (с экранированием `""`), скобки, идентификаторы.
2. **Парсер**: рекурсивный спуск по S-выражениям 
3. **Компилятор** (`utils/compiler.py`): обход AST, генерация инструкций для
   стековой машины. Адреса переменных/строк/массивов записываются временно как
   отрицательные смещения данных и доращиваются в `finalize()` до абсолютных
   (`n_instr + offset`); прямые ссылки на функции разрешаются там же.
   Литералы (строки, `LIST`) размещаются в секции данных, на стек кладётся адрес.

## Модель процессора

Консольное приложение `machine/processor.py`.

### Интерфейс командной строки:

```text
python -m machine.processor <binary.bin> [--input <строка|файл>] [--log] [--max-log N]
```

* Входные данные:
  * `binary.bin` - имя входного файла, с бинарным машинным кодом для запуска
  * `<строка|файл>` - можно указать имя файла или строку в формате "..." для передачи параметров на вход
  * `[--log]` - требуется ли вывод логов
  * `[--max-log N]` - ограничение на вывод логов
* Выходные данные:
    * `stdout` - производится лог работы программы (если параметр был указан), а так же выводится статистика: количество тактов, количество инструкций, а так же вывод, который был сделан
    * в `stdout` будет написан результат работы компилятора (сколько строк скомпилировано, название выходного бинарного файла и дамп)

#### Пошаговое выполнение инструкций

Процессор модулируется с точностью до тактов, каждая инструкция выполняется за различное число тактов. Каждая команда проходит через 2 стадии:
1. **Fetch** - 2 такта
   * На первом такте загружаем значение из `PC` в `AR`
   * На втором такте загружаем из памяти в `IR` команду, делаем инкремент `PC` (4 байта)
2. **Execute** - переменное число тактов
    * декодирование команды из `IR`
    * выполнение операции пошагово
    * количество тактов зависит от инструкции (не более 5, ниже будет расписано в таблице)

#### Ввод-вывод
* Ввод-вывод происходит через MMIO, т.е. стандартными инструкциями загрузки и записи
* Чтение происходит из буфера, без прерываний, при обращении к буферу получаем один байт


### DataPath

Горизонтальный микрокод: каждое поле управляющего слова идёт прямо на свой
узел. Стек данных умеет только push/pop, а что грузится в вершину — решают
мультиплексоры.

![Схема DataPath](photos/datapath.svg)

Главные на этой схеме стеки (DS, RS)
1. **ALU**
    * Управляющий код операции - `alu_op`
    * Правый вход — `TOS` из `DS`; левый вход выбирает мультиплексор `alu_sel`:
      `NOS` из `DS` либо константный `0` (последнее используется для 
      проверки на ноль в `JZ`: `0 − TOS`)
    * Выходные данные идут на MUX для записи в `DS` (`TOS`), а перенос/заём —
      на вход регистра `C`
2. **C**
    * Управляющий код - `c_latch` (разрешение защёлкивания), сам сигнал даёт CU
    * Регистр защёлкивает перенос/заём с выхода ALU, когда `c_latch = 1`
3. **DS**
   * Стек данных, хранит в себе значения во время выполнения программы
   * Управляется сигналами `ds_latch, dsp_delta` (первый отвечает за разрешение записать данные на стек, второе управляет DSP)
   * На выход отдаёт 2 своих верхних значения (`TOS, NOS`)
   * На вход можно записать только в `TOS`, стек сам положит старый `TOS` в `NOS`
4. **RS**
   * Стек возвратов, хранит в себе адреса возвратов для выхода из функции
   * Управляется сигналами `rs_latch, rsp_delta` (первый отвечает за разрешение записать данные на стек, второе управляет RSP)
   * На выход отдаёт только свой верхний элемент
   * На вход можно записать только в вершину стека
5. **MUX перед входом в DS**
    * Управляется сигналом - `TOS_sel`
    * Вход: 
        * Результат ALU
        * `NOS`
        * Значение из памяти
        * Аргумент инструкции
    * Выход идёт в TOS
6. **MUX перед AR**
    * Вход
      * `TOS` из `DS`
      * Аргумент инструкции
      * Значение `PC`
7. **MMIO**
    * MMIO - на входе стоит декодер, который определяет куда будет обращаться процессор, на выходе стоит мультиплексор, чтобы выбирать нужные данные

### ControlUnit (микропрограммный)

#### Вход/Выход

На вход в CU приходят первые 8 бит инструкции (`opcode`), а так же `C`-флаг

На выход из CU идут все управляющие сигналы, который нужны для `Datapath`


#### Внутренние блоки
1. **Dispatcher**
    * Хранит соответствие опкодов и адресов микроинструкций 
    * На вход получает опкод, на выход даёт адрес этой команды в памяти микрокоманд 
2. **Microcode ROM**
    * Хранит непосредственно микроинструкции (в начале лежам микроинструкции Fetch, чтобы он всегда исполнялся первым)
    * на вход получает адрес микроинструкции, на выход выставляется горизонтальный опкод, который сразу идёт в `Datapath`
3. **Sequencer**
    * Управляет выбором MUX, который отвечает за адреса микрокоманд
    * На вход принимает `C`-флаг и саму микрокоманду, чтобы понять какой переход совершать дальше
4. **MUX**
    * Управляет выбором MPC, можем перейти на следующую команду, или сделать переход по аргументу микрокоманды, или переход на другую микроинструкцию

#### Формат микрокоманды (горизонтальный микрокод)

Микрокод — **горизонтальный**: каждое поле управляющего слова идёт напрямую на
свой узел `DataPath` за один такт. Одно слово — 34 бита
(биты 0–33).

| Поле | Биты | Ширина | Значения | Назначение |
|------|:----:|:------:|----------|------------|
| `next_addr` | 0–5 | 6 | 0–63 | Адрес перехода в ПЗУ микрокода (для `NEXT`/`IF_C`/`IF_NC`) |
| `cond` | 6–8 | 3 | `SEQ`=0, `NEXT`=1, `DISPATCH`=2, `IF_C`=3, `IF_NC`=4 | Как секвенсор выбирает следующий MPC: +1 / по `next_addr` / по опкоду / по флагу `C` |
| `alu_op` | 9–12 | 4 | `ADD`=0, `SUB`=1, `MUL`=2, `DIV`=3, `MOD`=4, `EQ`=5, `LT`=6, `GT`=7, `ADDC`=8, `SUBB`=9 | Операция АЛУ |
| `ar_sel` | 13–14 | 2 | `PC`=0, `ARG`=1, `DS`=2 | Мультиплексор источника `AR` |
| `tos_sel` | 15–17 | 3 | `HOLD`=0, `NOS`=1, `MEM`=2, `ARG`=3, `ALU`=4 | Мультиплексор входа вершины `DS` (`TOS`) |
| `dsp_delta` | 18–19 | 2 | `HOLD`=0, `INC`=1, `DEC`=2 | Изменение указателя стека данных (DSP) |
| `ds_latch` | 20 | 1 | 0/1 | Разрешение записи в стек-память `DS` |
| `rsp_delta` | 21–22 | 2 | `HOLD`=0, `INC`=1, `DEC`=2 | Изменение указателя стека возвратов (RSP) |
| `rs_latch` | 23 | 1 | 0/1 | Разрешение записи в стек-память `RS` |
| `pc_sel` | 24–25 | 2 | `INC`=0, `ARG`=1, `RS`=2 | Мультиплексор источника `PC` |
| `pc_latch` | 26 | 1 | 0/1 | Защёлкнуть `PC` |
| `mem_latch` | 27 | 1 | 0/1 | Запись в память данных (`MEM[AR] ← TOS`) |
| `ir_latch` | 28 | 1 | 0/1 | Защёлкнуть `IR` — выборка команды |
| `halt` | 29 | 1 | 0/1 | Остановить тактовый генератор |
| `ns_latch` | 30 | 1 | 0/1 | Записать `NOS ← TOS` (для `SWAP`) |
| `c_latch` | 31 | 1 | 0/1 | Защёлкнуть флаг `C` из переноса/заёма АЛУ |
| `ar_latch` | 32 | 1 | 0/1 | Защёлкнуть `AR` |
| `alu_sel` | 33 | 1 | `NOS`=0, `ZERO`=1 | Мультиплексор левого входа АЛУ: `NOS` из `DS` либо константный `0` |

### Особенности моделирования

- **Потактовое** (`tick`): каждый такт = одно слово микрокода; журнал пишется
  по тактам (регистры, `MPC`, снимок `DS`/`RS`, исполняемая инструкция/микрооп).
- Ввод-вывод **потоковый**: `READ`(=`LOAD IN_PORT`) снимает токен из буфера,
  пустой буфер → `StopSignal` (останов). `WRITE`(=`STORE OUT_PORT`) дописывает
  байт в буфер вывода.
- В конце журнала — финальный дамп: причина останова, `output_buffer` списком
  байт, число тактов/инструкций, дамп памяти данных.

## Тестирование

Тесты сделаны с помощью pytest-golden, каждый из них представлен .yml файлом в директории `golden/`, исходные коды находятся в директории `sources/`

### Краткое описание разработанных тестов

| Тест | Что проверяет | Ввод → вывод |
|------|---------------|--------------|
| [hello](golden/hello.yml) | печать строкового литерала (cstr) через пользовательскую `PRINT-STR` (цикл `PEEK`+`WRITE`) | — → `Hello, World!` |
| [cat](golden/cat.yml) | потоковый ввод-вывод, останов по концу буфера (вход потенциально бесконечен) | `foo bar\nbaz` → то же |
| [hello_user_name](golden/hello_user_name.yml) | вывод приглашения, чтение имени до `\n`, приветствие (имя стримится в вывод без буферизации) | `Alice` → `What is your name?` / `Hello, Alice!` |
| [sort](golden/sort.yml) | сортировка списка чисел, загруженного как литерал-массив (`LIST`, по аналогии с cstr); read-only selection-сортировка через `PEEK` (стабильна, дубликаты) | литерал `5 3 8 1 3 9 2 7` → `1 2 3 3 5 7 8 9` |
| [double_precision](golden/double_precision.yml) | арифметика повышенной точности **на аппаратном переносе процессора**: число хранится лимбами по основанию 2³², сложение идёт `(+ …)` для младшего лимба и `(+C …)` для старших — перенос между лимбами считает АЛУ (флаг `C`), а не Lisp-сравнение | — → `7 31 4` (лимбы суммы `7·2⁶⁴ + 31·2³² + 4`; младший лимб `0xFFFFFFFF+5` переполнился → перенос в средний) |
| [prob2](golden/prob2.yml) | алгоритм по варианту — Найдите разницу между суммой квадратов первой ста натуральных чисел и квадратом суммы. | — → `25164150` |
| [factorial](golden/factorial.yml) | особенность языка: рекурсия (`CALL`/`RET`, сохранение параметра на стеке возвратов) и `IF` как выражение | — → `5040` (= 7!) |
